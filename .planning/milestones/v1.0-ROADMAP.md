# Milestone v1.0: MVP

**Status:** ✅ SHIPPED 2026-04-03
**Phases:** 1–7
**Total Plans:** 16

## Overview

A focused, 7-phase build: stand up the Rust project foundation with config parsing and PZEM-016 data types first, then build and validate the InfluxDB 3 write path, then integrate real Modbus RTU hardware and wire the full polling loop with error handling, then package the daemon for production deployment on Raspberry Pi via systemd, write a comprehensive E2E manual, add daily energy reset scheduling, and finally harden the daemon for indefinite production operation.

## Milestone Summary

**Key Decisions:**

- Rust for implementation — low memory footprint on Pi, reliability, strong serial/async ecosystem
- TOML config format — idiomatic in Rust ecosystem (serde + toml crate), human-friendly
- tokio current_thread runtime — single RS485 bus needs sequential polling, eliminates Send bounds on serial handles
- Per-device InfluxDB measurement (named) — allows per-device dashboards and queries without tag filtering
- tokio-modbus 0.17 + rtu::attach(port) — only async RTU crate integrating with tokio-serial; slave switched via set_slave()
- {:.4} float formatting for all PZEM fields — prevents InfluxDB 3 integer type lock-in (STOR-03)
- reqwest rustls feature (not rustls-tls) — avoids OpenSSL system library dependency during cross-compilation
- far_future() parks reset_sleep arm when disabled — no conditional select! needed
- CRIT-02 uses exit+systemd-restart (not in-process serial reconnect) — simpler, more reliable

**Issues Resolved:**

- All 14 daemon reliability findings addressed in Phase 7
- Daily energy reset with IANA timezone support via chrono-tz
- Log rotation (daily rolling) to prevent unbounded log growth on Pi SD card
- Config secret hygiene: config.toml removed from git, config.toml.example added
- InfluxDB HTTP timeouts: 5s connect + 10s request

**Issues Deferred (Tech Debt):**

- CFG-02 parity field missing from SerialConfig — PZEM-016 8N1 default works implicitly; add `parity: Option<String>` in v1.1
- RISK-1: udev rule uses `ch341` driver vs `cp210x` in comment/README — documentation mismatch, fix in v1.1
- STOR-01–STOR-04 checkboxes stale in REQUIREMENTS.md — code fully implemented and verified; only docs
- 5 phases missing VERIFICATION.md — phases 1, 2, 4, 5, 6 were not formally verified via /gsd-verify-phase
- REL-01 orphaned REQ-ID — referenced in ROADMAP.md but not defined in REQUIREMENTS.md

**Technical Debt Incurred:**

- reqwest feature `"rustls"` should be `"rustls-tls"` per canonical crates.io name — works today as alias
- 5 of 7 phases have no VERIFICATION.md — execution predated /gsd-verify-phase workflow
