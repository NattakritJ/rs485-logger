# Plan 07-03: Runtime Resilience — Serial Recovery + Modbus Safety + InfluxDB Backoff

## Findings Addressed

- **CRIT-02**: Serial port failure is unrecoverable — daemon must exit for systemd restart
- **HIGH-04**: Modbus context may enter broken state after transport error
- **MED-04**: No backoff on repeated InfluxDB failures (log spam)

## Tasks

### T1: Add consecutive all-device failure counter — exit for systemd restart (CRIT-02)

**File:** `src/main.rs` — poll loop tick arm

Track consecutive polls where ALL devices fail. If the counter reaches a configurable threshold (default 10), log an error and exit with code 1. systemd `Restart=always` will restart the process, re-opening the serial port.

```rust
let mut consecutive_all_fail: u32 = 0;
const MAX_CONSECUTIVE_ALL_FAIL: u32 = 10;

// In tick arm:
let mut any_ok = false;
for device in &cfg.devices {
    match poller.poll_device(device).await {
        Ok(reading) => {
            any_ok = true;
            // ... write to InfluxDB
        }
        Err(e) => { /* ... log warning */ }
    }
}
if any_ok {
    consecutive_all_fail = 0;
} else {
    consecutive_all_fail += 1;
    tracing::warn!(
        consecutive_failures = consecutive_all_fail,
        "All devices failed this poll cycle"
    );
    if consecutive_all_fail >= MAX_CONSECUTIVE_ALL_FAIL {
        tracing::error!(
            consecutive_failures = consecutive_all_fail,
            "All devices failed {} consecutive polls — exiting for systemd restart",
            MAX_CONSECUTIVE_ALL_FAIL
        );
        break;
    }
}
```

### T2: Add inter-device delay after timeout (HIGH-04)

**File:** `src/main.rs` — poll loop, after each failed `poll_device`

When a device poll times out (the most common cause of stale Modbus buffer data), add a short delay before polling the next device to let any late response drain from the serial buffer:

```rust
Err(e) => {
    tracing::warn!(...);
    // HIGH-04: After timeout/error, allow stale Modbus frames to drain
    tokio::time::sleep(std::time::Duration::from_millis(100)).await;
}
```

This 100ms delay is only added on errors (not on successful polls), so it doesn't impact normal throughput.

### T3: InfluxDB failure state tracking — suppress repeated log spam (MED-04)

**File:** `src/main.rs` — poll loop InfluxDB write section

Track whether InfluxDB was healthy on the last write. On transition from healthy to unhealthy, log a WARN. On transition from unhealthy to healthy, log an INFO. While unhealthy, suppress individual write failure logs.

```rust
let mut influx_healthy = true;

// In write section:
if let Err(e) = writer.write(&reading).await {
    if influx_healthy {
        tracing::warn!(error = %e, "InfluxDB write failed — suppressing further warnings until restored");
        influx_healthy = false;
    }
} else {
    if !influx_healthy {
        tracing::info!("InfluxDB connection restored");
        influx_healthy = true;
    }
}
```

### T4: Verify with cargo test + clippy

Run `cargo test` and `cargo clippy -- -D warnings` to ensure no regressions.

## Notes

- CRIT-02 uses the "exit and let systemd restart" approach (Option A from the report). This is the simplest and most reliable solution. Option B (reconnect) would require `ModbusPoller::reconnect()` which is more complex and harder to test without hardware.
- HIGH-04 delay is conservative (100ms). Can be tuned with hardware testing. The delay only fires on error, not on every device.
- MED-04 state tracking is per-daemon, not per-device. Since all devices write to the same InfluxDB, a single health flag is sufficient.
