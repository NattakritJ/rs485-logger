### 4. Hardware: PZEM-016 Wiring
This is the most important physical section. Cover:

**4.1 PZEM-016 terminal overview**
- Power input terminals: L and N — connect to the AC line you want to measure (120V/240V, single-phase). The device needs AC power to operate.
- Current transformer (CT) clamp: clips around the live wire (L) only. Does NOT break the circuit. Arrow on CT must point away from the source (toward the load).
- RS485 terminals (A/B): differential pair. A = D+ (positive), B = D− (negative).

**4.2 USB-RS485 adapter wiring**
Typical adapter pinout (varies by adapter):
```
PZEM-016 RS485-A (D+) ── Adapter A/D+
PZEM-016 RS485-B (D−) ── Adapter B/D−
GND (if present)        ── GND (optional but recommended for noise)
```
Note: Some adapters label terminals as A/B, others as D+/D−, T+/T−, or R+/R−. They are the same differential pair. Polarity matters: if the device doesn't respond, swap A and B.

**4.3 Daisy-chaining multiple PZEM-016 devices**
All devices share the same A and B lines in parallel:
```
Pi USB ── [Adapter] ── A/B bus ── [PZEM #1 addr=1] ── [PZEM #2 addr=2] ── [PZEM #3 addr=3]
```
Each device must have a unique Modbus address (1–247). The PZEM-016 ships with address 1. Use a Modbus address-change tool (e.g., the PZEM Windows software or a separate script) to assign addresses 2, 3, etc. before wiring them in parallel.

**4.4 Termination**
For RS485 runs longer than ~1m, add a 120Ω termination resistor across A and B at the far end of the bus. Most short bench setups work fine without it.

**4.5 Connecting the adapter to Raspberry Pi**
Plug the USB-RS485 adapter into any USB port on the Pi. The kernel will assign it `/dev/ttyUSB0` (or `/dev/ttyUSB1`, etc.). Run the udev rule (Section 7) to get a stable `/dev/ttyRS485` symlink.

To identify which port was assigned:
```bash
ls /dev/ttyUSB*
dmesg | tail -20   # look for "cp210x" / "ch341" / "ftdi_sio" + port assignment
```

### 7. Installation

**7.1 Build options**

*Option A — Native build on the Raspberry Pi (recommended for simplicity):*
```bash
# On the Pi:
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source ~/.cargo/env
git clone https://github.com/YOUR_REPO/rs485-logger.git
cd rs485-logger
cargo build --release
# Binary: target/release/rs485-logger
```

*Option B — Cross-compile from x86 Linux/Mac (faster iteration):*
```bash
# On your dev machine:
cargo install cross --git https://github.com/cross-rs/cross
# Pi 4/5 or Pi 3 (64-bit OS):
./deploy/build-release.sh
# Pi 2/3 (32-bit OS):
TARGET=armv7-unknown-linux-gnueabihf ./deploy/build-release.sh
# Binary: target/aarch64-unknown-linux-gnu/release/rs485-logger
```

**7.2 Deploy and install**
```bash
# Copy to Pi (cross-compile case):
scp target/aarch64-unknown-linux-gnu/release/rs485-logger pi@<PI_IP>:~/
scp -r deploy pi@<PI_IP>:~/deploy/

# SSH in and run installer (requires root):
ssh pi@<PI_IP>
sudo ~/deploy/install.sh ~/rs485-logger
```
The install script:
1. Creates a `rs485logger` system user (no login shell)
2. Adds it to the `dialout` group (serial port access)
3. Installs binary to `/usr/local/bin/rs485-logger`
4. Creates `/etc/rs485-logger/` (config directory) and `/var/log/rs485-logger/` (log directory)
5. Installs and enables the systemd service
6. Installs udev rule for `/dev/ttyRS485`

**7.3 Configure**
```bash
sudo cp config.toml /etc/rs485-logger/config.toml
sudo chmod 600 /etc/rs485-logger/config.toml
sudo chown rs485logger:rs485logger /etc/rs485-logger/config.toml
```

### 10. udev Rule (Stable Device Path)

The file `deploy/99-rs485.rules` creates a `/dev/ttyRS485` symlink. The default rule targets `cp210x` (SiLabs — the most common cheap USB-RS485 adapter chip).

**Finding your adapter's driver:**
```bash
udevadm info -a -n /dev/ttyUSB0 | grep -E 'DRIVERS|idVendor|idProduct'
```
Look for the `DRIVERS` line in the `usb` subsystem block: typically `cp210x`, `ch341`, or `ftdi_sio`.

If your adapter uses a different driver, edit `deploy/99-rs485.rules` and change `DRIVERS=="cp210x"` accordingly before running `install.sh`.

**Apply rule changes without reboot:**
```bash
sudo udevadm control --reload-rules && sudo udevadm trigger
ls -la /dev/ttyRS485   # symlink should appear
```

### 11. Troubleshooting

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| `Failed to open serial port '/dev/ttyRS485'` | Adapter not plugged in, or udev rule not loaded | Check `ls /dev/ttyUSB*`; reload udev rules |
| `Failed to open serial port '/dev/ttyRS485'` (permission) | `rs485logger` user not in `dialout` group | `sudo usermod -aG dialout rs485logger` then restart |
| `Timeout polling device 'X'` | Wrong baud rate, wrong address, wiring reversed | Verify `baud_rate = 9600`, check device address, try swapping A/B wires |
| `InfluxDB write failed: HTTP 401` | Wrong or expired API token | Regenerate token in InfluxDB UI, update config.toml |
| `InfluxDB write failed: connection refused` | InfluxDB not running or wrong URL | Check InfluxDB service status; verify `influxdb.url` in config |
| Daemon crashes on startup | Config parse error | Run `rs485-logger --config /etc/rs485-logger/config.toml` manually to see the error message |
| All devices show `Timeout` but wiring looks correct | 32-bit word order mismatch | PZEM-016 uses low-word-first; if register decode looks wrong, values may be hardware-dependent |
| No data after reboot | systemd unit not enabled | `sudo systemctl enable rs485-logger` |

### 12. PZEM-016 Register Map (Reference)
Compact table for users who want to understand the data:

| Register | Field | Scale | Unit |
|----------|-------|-------|------|
| 0x0000 | Voltage | ÷10 | V |
| 0x0001–0x0002 | Current (low-word-first 32-bit) | ÷1000 | A |
| 0x0003–0x0004 | Power (low-word-first 32-bit) | ÷10 | W |
| 0x0005–0x0006 | Energy (low-word-first 32-bit) | ×1 | Wh |
| 0x0007 | Frequency | ÷10 | Hz |
| 0x0008 | Power Factor | ÷100 | — |
| 0x0009 | Alarm status | — | ignored |

Note: 32-bit values use low-word-first byte order (PZEM-016 deviation from Modbus standard).
