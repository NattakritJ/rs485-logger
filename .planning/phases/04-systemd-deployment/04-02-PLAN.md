  <name>Task 1: Cross.toml configuration</name>
  <files>Cross.toml</files>
  <action>
Create `Cross.toml` at the project root to configure `cross` for both Raspberry Pi targets.

**Important constraint:** `tokio-serial` (via `libudev`) requires a system library. The cross Docker images for arm targets include `libudev-dev` in the `aarch64-unknown-linux-gnu` image by default, but we should explicitly declare it to avoid surprises.

```toml
# Cross.toml — configuration for cargo-cross cross-compilation
# Install cross: cargo install cross --git https://github.com/cross-rs/cross
# Usage: cross build --target aarch64-unknown-linux-gnu --release
#        cross build --target armv7-unknown-linux-gnueabihf --release

[build.env]
# No additional env vars needed — reqwest uses rustls (no OPENSSL_DIR required)

[target.aarch64-unknown-linux-gnu]
# Raspberry Pi 4, Pi 5, Pi 3 running 64-bit OS (Raspberry Pi OS Bookworm 64-bit)
pre-build = [
    "dpkg --add-architecture arm64",
    "apt-get update",
    "apt-get install -y --no-install-recommends libudev-dev:arm64",
]

[target.armv7-unknown-linux-gnueabihf]
# Raspberry Pi 2, Pi 3 running 32-bit OS (Raspberry Pi OS Bookworm 32-bit)
pre-build = [
    "dpkg --add-architecture armhf",
    "apt-get update",
    "apt-get install -y --no-install-recommends libudev-dev:armhf",
]
```

Add a comment block above each target section explaining which Raspberry Pi hardware/OS it targets.

**Why no OPENSSL changes:** `reqwest` was locked to `features = ["rustls"]` in Phase 1 (D-01 in STATE.md) precisely to avoid needing OpenSSL during cross-compilation. No `OPENSSL_DIR` or `OPENSSL_STATIC` env vars required.

<task type="auto">
  <name>Task 2: build-release.sh wrapper + cross install check</name>
  <files>deploy/build-release.sh</files>
  <action>
Create `deploy/build-release.sh` — a script that cross-compiles the release binary, checks prerequisites, and prints the output path.

```bash
#!/usr/bin/env bash
set -euo pipefail

# Default target: Raspberry Pi 4/5 (aarch64, 64-bit OS)
# Override: TARGET=armv7-unknown-linux-gnueabihf ./deploy/build-release.sh
TARGET="${TARGET:-aarch64-unknown-linux-gnu}"

echo "=== rs485-logger release build ==="
echo "Target: $TARGET"
echo ""

# Check cross is installed
if ! command -v cross &>/dev/null; then
    echo "ERROR: 'cross' not found."
    echo "Install: cargo install cross --git https://github.com/cross-rs/cross"
    exit 1
fi

# Check Docker is running (cross requires it)
if ! docker info &>/dev/null 2>&1; then
    echo "ERROR: Docker is not running. cross requires Docker."
    exit 1
fi

# Build
cross build --target "$TARGET" --release

BINARY="target/$TARGET/release/rs485-logger"

echo ""
echo "=== Build complete ==="
echo "Binary: $BINARY"
echo "Size:   $(du -sh "$BINARY" | cut -f1)"
echo ""
echo "Deploy steps:"
echo "  1. Copy binary to Pi:    scp $BINARY pi@<PI_IP>:~/rs485-logger"
echo "  2. Run install script:   ssh pi@<PI_IP> 'sudo ~/deploy/install.sh ~/rs485-logger'"
```

Make executable: `chmod +x deploy/build-release.sh`

Then attempt the actual cross-compilation to verify it builds without linker errors:

```bash
# Install cross if not present (uses --locked for reproducibility)
cargo install cross --git https://github.com/cross-rs/cross --locked 2>/dev/null || true

# Attempt cross-build (requires Docker; if Docker unavailable, verify Cargo.toml config is correct instead)
if command -v cross &>/dev/null && docker info &>/dev/null 2>&1; then
    cross build --target aarch64-unknown-linux-gnu --release
    echo "Cross-compilation: SUCCESS"
    ls -lh target/aarch64-unknown-linux-gnu/release/rs485-logger
else
    echo "cross/Docker not available — verifying native build instead"
    cargo build --release
    echo "Native release build: SUCCESS"
    ls -lh target/release/rs485-logger
fi
```

If cross-compilation fails due to libudev linking, the fallback is native compilation on the Raspberry Pi itself (`cargo build --release` after `rustup target add aarch64-unknown-linux-gnu`). Document this in a comment in the script.
